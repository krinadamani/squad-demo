- Developer: Building on the Lead's block, here's the developer assessment:

**Root cause areas:** The polling loop in `watch/index.ts` uses a single boolean (`roundInProgress`) as a concurrency guard with no timeout — if `executeRound()` hangs, subsequent intervals silently skip without alerting. Combined with `monitor.stop()` having no timeout in the shutdown sequence, a stalled shutdown causes the 3s signal handler window in `cli-entry.ts` to expire mid-cleanup, leaving the process in a dirty state. That's the most likely source of intermittent CI failures.

**Test harness gap:** `watch.test.ts` only validates module exports and `reportBoard` output — it never exercises `runWatch()`, the polling interval, or shutdown. Flakiness is invisible to the current test suite because the failure mode (hung async teardown) only surfaces under real process lifecycle conditions.

**Safe fix strategy:** (1) Replace the boolean race guard with a proper async mutex or serial queue. (2) Add a `Promise.race()` timeout (e.g. 5s) around `monitor.stop()` and `eventBus.emit()` in the shutdown handler. (3) Bump the signal handler timeout in `cli-entry.ts` from 3s to 10–15s specifically for watch mode. (4) Add at minimum one integration test that spawns `runWatch()` and asserts clean exit on SIGTERM within a bounded time.

**Rollback plan:** The three changed files (`watch/config.ts`, `cli-entry.ts`, `watch-mode.test.ts`) are all independently revertible. If stabilization stalls, revert `cli-entry.ts` and `watch/config.ts` to HEAD~1 on `dev`, quarantine the watch-mode test under a `skip`/`xit` with a tracking issue, and let the PR sit until root cause is confirmed on a feature branch — no partial merge.
